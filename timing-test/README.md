# Cross-emulator CPU & bus timing test

A tiny bootable Amiga disk that measures how long a fixed amount of CPU and
memory work takes, using the CIA-A timer (E-clock) as the reference, and shows
the raw results on screen as hex (and streams them out the serial port). Run the
same disk on Copperline, vAmiga and FS-UAE and compare: any number that differs
tells you exactly which operation an emulator times differently.

A small boot block (`boot.asm`) loads the main program (`test.asm`) to a fixed
address ($30000) and jumps to it; the main program takes over the machine
completely (interrupts and DMA off, no OS calls), so the measurement is not
perturbed by the system ROM. Loading to a fixed address also matters: the ROM's
boot buffer can land in slow RAM, and code-fetch timing must not depend on where
the code happens to sit.

## Building

Needs [vasm](http://sun.hasenbraten.de/vasm/) (`vasmm68k_mot`, Motorola syntax):

```sh
make CPU=m68k SYNTAX=mot      # in the vasm source tree
./build.sh                    # then here (set VASM=/path/to/vasmm68k_mot)
```

`timing-test.adf` is generated by `./build.sh` and is intentionally
gitignored; rebuild it locally before running the checks.

## Running

- **Copperline:** `copperline --config timing-test.toml --screenshot-after 12 out.png`
  Serial output also appears on stdout (one 8-digit hex value per line).
- **FS-UAE:** open `timing-test.fs-uae` (A500, 512K chip + 512K slow, PAL).
- **vAmiga:** A500, OCS, PAL, 512K chip + 512K slow (trapdoor) RAM, KS 2.05;
  insert `timing-test.adf` in DF0 and boot.

Use the same machine on all three: **OCS, PAL, 512K chip + 512K slow RAM**, so
the slow-RAM ($C00000) tests are meaningful.

### A1200 (no slow RAM)

The disk also boots on a machine with **no slow RAM** -- e.g. a real A1200
(68EC020, AGA, 2 MB chip). The four slow-RAM rows (0, 1, 9, 13) probe $C00000
at start-up and, when it is absent, store a `00000000` sentinel instead of
reading or executing unmapped space (row 13's `jsr $C00000` would otherwise
trap). Every other row -- including the copper-vs-CPU row 27, which is what the
020 chase-the-beam effects depend on -- runs normally. Copperline configs:
`tt-a1200.toml` (KS 3.1, AGA, 2 MB chip, no slow) and `tt-020-noslow.toml`
(68EC020, ECS, 2 MB chip, no slow, KS 2.05); compare against FS-UAE/vAmiga set
up the same way.

## What each row measures

Most values are the elapsed CIA E-clock ticks for the test (8192 iterations
for the fixed-count rows); rows 19, 20 and 22 are raw VHPOSR beam positions
and rows 23-26 are beam color-clock counts. A faithful emulator models a
fixed CPU-cycle / E-clock ratio, so identical CPU work should give identical
ticks everywhere.

| Row | Test | Instruction |
|-----|------|-------------|
| 0 | slow-RAM read   | `move.w (a0),d0`  a0=$C00000 |
| 1 | slow-RAM write  | `move.w d1,(a0)`  a0=$C00000 |
| 2 | chip-RAM read   | `move.w (a0),d0`  a0=$060000 |
| 3 | chip-RAM write  | `move.w d1,(a0)`  a0=$060000 |
| 4 | register move   | `move.w d2,d0` (4 cycles) |
| 5 | shift           | `lsl.l #8,d3` (24 cycles on a real 68000) |
| 6 | multiply        | `mulu #$5555,d5` (~54 cycles on a real 68000) |
| 7 | loop baseline   | `dbra` only (10 cycles taken) |
| 8 | frame length    | E-clock ticks for one whole video frame |
| 9 | slow reads/frame | slow reads completed from frame top to vpos 280 |
| 10 | chip write x1024 | `move.w d1,(a0)` 1024 times, no display DMA |
| 11 | chip write x1024 during 6-plane DMA | same, while a heavy lores bitplane display is fetching |
| 12 | chip write x1024 during 8-sprite DMA | same, while 8 sprites are DMA-fetching |
| 13 | dbra x8192 from slow RAM | the loop's code is executed from $C00000 |
| 14 | dbra x8192 from chip RAM | the same loop executed from $060000 |
| 15 | chip write x1024 during 6-bitplane + 8-sprite DMA | combined contention |
| 16 | chip writes/frame, no interrupts | per-frame throughput baseline |
| 17 | chip writes/frame, VERTB interrupt each frame | 16-17 = interrupt cost |
| 18 | chip write x1024 during 3-bitplane DMA | does contention scale with plane count? |

| 20 | beam (VHPOSR) at end of the chained SOFTINT task switch | 20-19 = the chain's raster cost |
| 21 | chip writes/frame with VERTB+SOFTINT+task-switch each frame | full per-frame chain cost |
| 22 | VHPOSR when the INTREQR VERTB bit first reads SET (interrupts off) | the pure VERTB-raise position |

| 26 | beam cck while the A->D fill (row 24) runs **with a 3-bitplane display + BLTPRI** | beam-vs-blitter race with active display DMA |
| 27 | VHPOSR when the CPU first sees a COPPER-raised INTREQR.COPER, interrupts off | copper-vs-CPU phase (`vp`<<8 | `hpos`/2); `FFFFFFFF` if it never fired |

### Row 27: the copper-vs-CPU interrupt phase

A copper list waits for line `$64`, MOVEs SET-COPER into INTREQ, then sleeps; the
CPU (interrupts off) busy-polls INTREQR for the COPER bit and records the beam
the moment it reads set. Decode it like rows 19/20 (high byte = vpos, low byte =
hpos/2). This is the mechanism SANITY Roots II's copper-chunky "cherries" screen
(and similar 020 effects) runs on: a copper-raised interrupt the per-frame work
spins on, then chases the beam to update the colour list before the copper reads
it. The value is the copper WAIT-release + MOVE + INTREQR-visibility delay +
poll-catch latency, so a difference across emulators is a copper-vs-CPU phase
divergence -- exactly what makes that effect render or scramble. (Copperline
reads `$6409` on an A1200 at 14 MHz, `$640C` at the 7 MHz reference clock.)

### Rows 23-26: the blitter-vs-beam race

None of rows 0-22 run an active blitter, yet whether a beam-raced blitter draw
keeps **ahead of the display beam** is what decides if a triple-buffered vector
effect shows or blanks: the buffer Agnus fetches for display must already be
finished. These rows record, in **raster color clocks** (`vpos*227 + hpos`), how
far the beam advances while one fixed blit shape runs to completion (polling
DMACONR BBUSY). Rows 23-25 have display DMA off (pure blitter cycle cost vs the
beam); row 26 reruns the area fill with a 3-bitplane display active and BLTPRI
set, so **row 26 - row 24 is the display-DMA contention cost on the blitter**.
The blitter and the beam share the chip clock, so a faithful emulator should
report the same cck everywhere; a difference is a blitter-timing-vs-beam bug --
the dimension the CPU/contention rows cannot expose.

These rows found (and now guard) a real bug: Copperline originally reported
rows 23-26 = 9944 / 15308 / 383 / 16842 cck, timing the row-24 A->D fill at
~2 cck/word where FS-UAE (10004 / 18364 / 344 / 25095) and vAmiga
(10070 / 18350 / 346 / 25097) agree on ~3 cck/word -- the too-fast fill let the
display fetch an unfinished buffer. The fill cadence was corrected on 2026-06-09;
Copperline now reads ~9922 / 18339 / 317 / 25074 against the current FS-UAE
reference 9908 / 18357 / 262 / 25208, matching rows 23-24 and the row-26
display-DMA contention. Known residual tracked by these rows: row 25 (line
blit) is ~21% slow (317 vs 262).

### Rows 19-21: the cooperative-scheduler interrupt chain

These target a chained interrupt pattern used by cooperative two-thread
software: every frame the VERTB (level 3) interrupt fires, its handler SETs a
level-1 SOFT request, and on the RTE the level-1 SOFT handler performs a **task
switch** (save all 15 data/address registers, swap the stack pointer, restore
registers) before control returns to the render loop. The loop top therefore
restarts a few raster lines *after* the vblank, and that startup latency eats the
per-frame budget. If an emulator advances the beam by even a line more than real
hardware across this fixed interrupt+switch sequence, a depacker/render slice can
tip just past the vblank and halve the effective frame rate. Rows 16/17 only
exercise a bare VERTB ack; they do **not** cover the SOFTINT + register-file task
switch, which is what rows 19-21 add.

- **Row 19** is the beam at the first instruction of the VERTB handler -- the
  interrupt-recognition latency measured from the top of the frame.
- **Row 20** is the beam at the end of the level-1 SOFT task-switch handler.
  **(Row 20 - Row 19) is the raster cost of the whole VERTB -> SET SOFT -> SOFT
  -> task-switch chain** -- the direct "how many lines does this chain burn"
  number. This is the most diagnostic of the three.
- **Row 21** is the same per-frame throughput measure as rows 16/17 but with the
  full chain firing each frame; row16 - row21 is the chain's cost in displaced
  writes (a low-resolution cross-check of rows 19/20).

### Row 22: VERTB raise position (raise vs. latency)

Rows 19/20 measure handler *entry* = the VERTB raise position **plus** the 68000
exception-entry latency. **Row 22 isolates the raise alone**: with interrupts
disabled it clears VERTB, then polls `INTREQR` in a tight loop and records the
beam (VHPOSR) the moment bit 5 reads set -- no exception in the path. Decode it
like rows 19/20 (high byte = vpos, low byte = hpos/2).

This disambiguates *where* a row-19 difference comes from:
- If **row 22 differs across emulators the same way row 19 does**, the VERTB
  **raise** is at a different beam position -- a chipset bug.
- If **row 22 matches but row 19 does not**, the raise is the same and the
  difference is **68000 interrupt latency** instead (a CPU-core matter).

**Outcome (2026-06-09):** row 22 matches across all three emulators
(Copperline `0x0008`, vAmiga `0x0007`, FS-UAE `0x000D` -- all vpos 0, a few cck
into the line), so the VERTB **raise** position is the same everywhere. The
row-19 entry difference was therefore **68000 interrupt-recognition latency**,
which Copperline was missing and now models (65 cck by default; override or
disable with `COPPERLINE_IRQ_LATENCY_CCK`). One test bug was also fixed along
the way: an earlier revision of this row read VHPOSR in the line-start
vpos-read-delay window at the frame wrap and reported a bogus ~`0x3870`
(vpos 56) on Copperline; the test now samples the beam only after the VERTB bit
reads set, which avoids that window.

**Decoding rows 19/20:** the value is a raw VHPOSR word -- the **high byte is the
vpos (raster line)** and the **low byte is hpos/2** (so hpos = low byte x 2; one
unit ~= 2 color clocks). With the interrupt-recognition-latency model in place,
Copperline reports row 19 = `0x003A` (line 0, hpos ~116), matching vAmiga
(`0x0039`) and FS-UAE (`0x003D`). Row 20 reads ~`0x0154` on Copperline vs
`0x0143`/`0x0147` on the references -- a known ~25 cck residual: the model
restarts the full latency for the SOFT interrupt raised mid-handler, where real
hardware overlaps it with the VERTB handler. Compare the raw words across
emulators: a higher row 19/20 means that emulator advances the beam further
across the interrupt + task-switch path -- enough extra latency can push a
per-frame loop top late and trip its frame skip.

## What to look for

The most diagnostic rows are **5 (shift)** and **6 (multiply)**: on a real 68000
they cost far more than a register move or a slow read, because of their
internal cycles. If an emulator reports row 5 ~= row 4, or row 6 ~= row 0, it is
not billing those internal cycles to the chipset clock.

Row 8 (frame length) should be ~the same on every emulator (the frame is defined
by the video beam, not the CPU). Row 9 (CPU work completed per frame) is the
direct "how much CPU fits in a frame" figure; a difference there is exactly the
kind of CPU-vs-beam mismatch that makes a frame-budget-sensitive demo behave
differently.

Rows 10-12 isolate chip-bus contention. Row 10 is chip writes with the display
off; rows 11 and 12 are the same writes while a heavy bitplane display (row 11)
or 8 sprites (row 12) are fetching, so the CPU competes with the DMA for the chip
bus. The **ratios row 11 / row 10** and **row 12 / row 10** are the contention
cost during a busy display; compare them across emulators.

Rows 13 and 14 check code-fetch location. The same loop is executed from slow RAM
(row 13) and chip RAM (row 14, matching row 7's chip-RAM dbra). With DMA off both
regions are 2 cck/word, so on real hardware rows 7, 13 and 14 are identical. If an
emulator times code differently depending on whether it runs from slow or chip
RAM, rows 13 and 14 diverge -- which matters for any program (like a demo) that
runs from slow/trapdoor RAM.
